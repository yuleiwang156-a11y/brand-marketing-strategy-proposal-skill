# 品牌营销战略建议书生成 Skill

> 版本：v2.0
> 适用对象：品牌战略/营销战略咨询团队、个人顾问、内部 Agent 工作流
> 输出形态：Word 文档，但内容按 PPT 逐页脚本输出
> 范围：只生成战略内容部分，不生成公司介绍、服务流程、报价、团队介绍等商务内容

> v2.0 主要变化：
> 1. 确认点 1 轻量化，默认按五项结构输出；
> 2. 不得在确认点、研究分析、机会洞察、目录生成、逐页建议书内容中，将‘三品破局 / 品牌破局 / 品类破局 / 产品破局 / 心智空位 / 工厂空位’作为判断框架、分析分类或客户可见输出表述使用。上述词语只允许出现在 Skill 规则的禁用说明中。
> 3. 机会点统一改为四维交叉推导：行业趋势 + 消费者需求 / 心智 + 竞品格局 + 企业自身优势 = 具体机会点；
> 4. 研究分析层重构（行业漏斗、消费者多维、竞对四项核心、企业自身扫描下移）；
> 5. 确认点 2 重构为“具体机会点洞察”，必须输出机会点交叉分析表；
> 6. 页数规则改为 50-60 页区间；
> 7. 逐页 PPT 内容脚本新增章节任务边界、信息释放顺序、资料依据层级和项目概况禁提前亮机会点规则。

# 辅助文件调用规则

本 Skill 除主文件外，还包含以下辅助文件。执行过程中必须按阶段读取并遵守。

## 1. 系统总览文件

执行任何新客户项目之前，必须读取：

- references/system-overview.md

用途：
用于理解本 Skill 的适用边界、核心流程、核心判断逻辑、输出原则，以及与通用品牌全案 Skill 的区别。

注意：
该文件是执行约束，不是客户资料，不得作为客户版建议书资料依据。

## 2. 逐页输出规范文件

进入逐页 PPT 内容脚本生成阶段前，必须读取：

- references/output-spec.md

用途：
用于约束逐页 PPT 脚本的页面格式、页面标题、页面正文、章节边界、信息释放顺序、资料依据规则和禁止事项。

注意：
逐页输出时，必须遵守该文件中的章节边界规则，尤其是：

- 项目概况不得提前输出机会点；
- 项目概况不得提前输出大单品候选；
- 环境分析不得提前输出客户最终机会点；
- 机会洞察前不得提前输出战略承接；
- 资料依据不得大量引用确认点、用户反馈、Skill 规则、MEMORY.md 或 agent_memory。

## 3. 研究分析阶段说明文件

进入研究分析层前，必须读取：

- instructions/research-analysis.md

用途：
用于规定行业趋势、消费者需求 / 心智、竞品格局、企业自身分析的执行方式。

注意：
研究分析层只提供证据基础，不得提前输出最终机会点、战略大单品候选或战略承接方案。

## 4. 机会点洞察阶段说明文件

进入确认点 2 / 具体机会点洞察前，必须读取：

- instructions/opportunity-insight.md

用途：
用于规定四维交叉机会点推导方式、机会点具体性要求、机会点压缩规则和机会点具体性检查表。

注意：
机会点必须具体到消费者能感知、能表达、能传播的词或方向，不得停留在“专业化、可信赖、有效、健康、高端、功能场景、制药级”等泛词。

## 5. 逐页生成阶段说明文件

进入逐页 PPT 内容脚本生成阶段前，必须读取：

- instructions/page-script-generation.md

用途：
用于规定逐页生成阶段的启动条件、分批输出规则、页面格式、章节边界、暂停确认规则和 Word 生成规则。

注意：
逐页脚本必须按客户可见一级目录分批输出，不得默认每 10 页一批输出。每部分完成后必须暂停等待用户确认。

## 6. 客户资料填写模板

当用户需要准备新客户资料时，可参考：

- templates/client-brief-template.md

用途：
用于指导团队成员整理客户基础信息、客户诉求、企业资料、产品资料、竞品资料、消费者资料、行业资料和交付要求。

注意：
该模板只是输入资料整理工具，不是客户事实依据，不得作为建议书正文资料来源。

references、instructions、templates 文件均不是客户资料。

不得把以下文件作为客户版建议书的资料依据：

- references/output-spec.md

- templates/client-brief-template.md

客户版建议书资料依据应优先来自：

- 客户资料；
- 品牌手册；
- 产品手册；
- 官网 / 官方资料；
- 行业报告；

- 竞品官网 / 旗舰店 / 公开资料；
- 研究分析层中的资料来源。

## 1. Skill 目标

本 Skill 用于帮助 Agent 基于客户行业、产品/服务、客户诉求、客户资料、行业与竞品公开信息，以及公司内部方法论，生成一份完整的**品牌营销战略建议书初稿**。

最终输出不是普通文章型 Word，而是：

> **用 Word 承载的 PPT 逐页内容脚本。**

最终 Word 文档不是过程记录，也不是 Agent 执行日志，而是给 PPT 制作方使用的页面脚本。最终 Word / PPT 制作版中每页只保留页码、页面标题和页面正文内容，不展示页面类型、页面正文标签、资料依据、资料依据标签或阶段判断标签。

过程审核版每一页都要明确：

- 页面类型
- 页面正文
- 资料依据

过程审核版用于对话中分批展示，方便用户审核 Agent 是否按规则执行；最终 Word / PPT 制作版必须清洗成只含页面稿。

本 Skill 的核心价值不是简单写作，而是将“客户诉求理解、问题定义、行业研究、消费者需求 / 心智分析、竞对分析、企业自身扫描、具体机会点洞察”变成可复用、可部署、可分享的 Agent 工作流。

---

## 2. 适用场景

本 Skill 适用于：

- 品牌营销战略建议书
- 品牌定位/营销战略方向建议书
- 品牌增长诊断类建议书
- 品牌机会点洞察类建议书

本版不适用于：

- 纯设计类建议书
- 纯活动策划类建议书
- 纯代运营类建议书
- 纯广告投放类建议书
- 只讲服务模块、报价、团队、项目周期的商务提案

如果客户真实诉求经过分析后不是品牌营销战略问题，Agent 不得强行套用品牌营销战略框架，而应围绕客户真实诉求重新搭建建议书逻辑。

## 3. 最终输出要求

### 3.1 输出形式

输出为 Word 文档内容，但 Word 内部必须按 PPT 页面脚本组织。

最终 Word 等同于“给 PPT 制作方使用的页面脚本”，只保留：

1. 页码；
2. 页面标题；
3. 页面正文内容。

最终 Word 不得包含：

- 页面类型；
- “页面正文：”标签；
- 资料依据；
- “资料依据：”标签；
- “阶段判断：”固定标签；
- Agent 过程说明；
- 内部审核说明；
- Skill 执行说明；
- 质量检查报告；
- 任何给用户看的操作提示。

最终 Word 也不得包含以下“生成过程控制信息”：

- 本批生成内容；
- 本部分对应客户可见一级目录；
- 本章任务；
- 本章不回答；
- 本章预计页面；
- 本部分到此暂停；
- 请确认是否通过；
- 通过后可回复；
- 当前阶段说明；
- Agent 执行说明；
- 质量检查报告；
- 总体检查结论；
- 是否通过 Skill 规则检查；
- 任何给用户看的操作提示。

这些内容可以在对话中出现，用于用户审核，但不得进入最终 Word。

过程审核版每页格式如下：

```markdown
# 第 X 页：页面标题

页面类型：
分析页 / 结论页 / 反问页 / 金句页 / 思考页 / 过渡页 / 概念解释页 / 案例页 / 目录页

页面正文：
写出这一页 PPT 上应该呈现的主要内容，包括标题、核心文字、要点、表格内容或图示说明。

资料依据：
标注本页内容依据。若涉及行业数据、市场规模、增速、份额、竞品事实，需要写明来源；如果暂时没有明确来源，需要标注“需人工补充出处”。
```

最终 Word / PPT 制作版每页格式如下：

```markdown
# 第 X 页：页面标题

页面正文内容……
```

这里的“页面正文内容”不写“页面正文：”四个字，只直接呈现未来 PPT 页面上要出现的内容。

过程审核版逐页内容必须绑定研究依据：

- 如果是客户资料中的事实，资料依据写明客户文件名。
- 如果是行业、消费者、竞对资料中的事实，资料依据必须引用研究依据库。
- 如果是 Agent 的战略判断，必须说明其推导依据来自哪些信息。
- 如果资料不足，必须在资料依据处标注“需人工确认”或“需补充出处”。
- 不得把没有证据支撑的判断写成确定结论。

资料依据层级规则：

1. 客户版逐页脚本的资料依据应优先引用：客户资料、品牌手册、官网 / 官方资料、行业报告、政策文件、消费者调研资料、竞品官网 / 旗舰店 / 公开资料、研究分析层中的资料来源。
2. 确认点 1、确认点 2、确认点 3 属于内部过程文件，只能作为内部生成依据，不应大量出现在客户版页面资料依据中。
3. 允许写“依据客户资料、品牌手册及前序研究分析。”不推荐写“依据确认点 2 用户反馈……”“依据确认点 3 已通过……”“依据项目 Skill 规则……”。
4. 客户版逐页脚本的资料依据不得主要依赖确认点，不能把用户反馈写成客户版资料依据，Skill 规则不能作为客户版页面资料依据。
5. 如果必须引用确认点，应转译为“前序研究结论”，并在内部使用，不作为客户可见资料来源。
6. MEMORY.md、agent_memory 或任何项目记忆文件，只能作为执行上下文使用，用于判断当前执行阶段、已完成步骤、之前测试中出现过的问题和用户已经确认过的流程规则；不得作为研究依据、客户资料依据、行业资料依据、消费者需求依据、竞对资料依据或建议书正文资料来源。
7. 在研究分析层、研究依据库、行业资料及来源、消费者需求 / 心智资料及来源、竞对资料及来源、企业自身资料归类、确认点 2 的四维交叉证据、逐页 PPT 内容脚本的资料依据和最终建议书正文中，不得引用 MEMORY.md、agent_memory 或任何项目记忆文件。
8. 如果 Agent 读取了 MEMORY.md，只能在内部执行时参考，不得把它列入“资料来源”或“引用依据”。如需说明，可写：“已参考项目执行记忆判断当前阶段，但该文件不作为研究依据。”

资料依据用于研究分析、过程审核版和内部检查，不进入最终 Word / PPT 制作版。如需保留资料依据，必须单独生成内部文件，例如 `outputs/资料依据清单.md`。

### 3.2 页数要求

- 完整建议书初稿原则上控制在 **50-60 页** PPT 内容。
- 少于 50 页说明论证不足；超过 60 页说明表达需要压缩或合并。
- 不得为了页数牺牲核心论证，也不得无限展开。
- 页数由客户问题复杂度、行业研究深度、核心竞品数量、机会点推导复杂度、是否涉及战略大单品分析共同决定，但总量须收敛到 50-60 页区间。
- 不允许只输出大纲，不允许把核心分析压缩成少数几页。
- 行业研究、消费者需求 / 心智分析、竞对分析、企业自身扫描、机会点洞察等核心板块必须充分拆页；如展开后超过 60 页，应通过合并相近页面、精简表达控制在区间内，而不是删除核心论证。

### 3.3 第一页要求

最终 Word 可以包含封面。若包含封面，第 1 页为封面，第 2 页为整体目录页；若不单独设置封面，第 1 页必须是整体目录页。

目录不能套固定模板，必须基于本次客户诉求和建议书主线生成。目录可以采用“大标题 + 小标题”的方式，让客户一开始就知道这份建议书要带他看什么。

### 3.4 默认展示、可选保存与长流程处理

Skill 的默认输出方式是在对话框中直接展示当前节点的完整可审核内容，并在展示后暂停等待用户确认。用户不需要打开本地文件，也应该能够直接判断当前确认点是否通过。

默认情况下：

1. 确认点 1 必须在对话框完整展示。
2. 研究依据库必须在对话框展示审核版内容。
3. 确认点 2 必须在对话框完整展示。
4. 确认点 3 必须在对话框完整展示。
5. 逐页 PPT 内容脚本必须分批在对话框展示，默认按确认点 3 已确认的“客户可见一级目录”逐章输出，每输出完一个一级目录对应页面后暂停等待确认。
6. 质量检查报告必须在对话框展示，或另存为独立 Markdown 文件；不得写入最终 Word。

`outputs` 文件夹只能作为可选工作缓存 / 可选阶段存档，不是强制流程。如果当前环境支持文件系统且用户明确要求保存，可保存到 `outputs` 或用户指定目录；否则只需在对话框输出。不得因无法保存 `outputs` 文件而停止任务。

如果任务较长、上下文可能溢出，Agent 应优先分阶段执行：每个确认点先在对话框展示并等待确认；逐页内容默认按客户可见一级目录分批生成；如某个一级目录过长，可在该目录内部按逻辑小节拆分；如必须新开线程，应让用户复制上一阶段确认结果或读取阶段文件。

### 3.4.1 逐页内容分批输出规则

确认点 3 通过后，Agent 开始生成 Word 版 PPT 逐页内容脚本。

逐页内容不得默认按“每 10 页一批”机械切分。默认分批方式应为：按照确认点 3 已确认的“客户可见一级目录”逐章输出。每进入一个一级目录前，应有目录 / 章节分隔页，并高亮当前章节。

章节分隔页用于展示完整目录、高亮当前章节、提醒客户当前汇报进度。章节分隔页正文不要写大量解释，只展示目录结构和当前章节。

例如客户可见一级目录为：项目概况 / 问题诊断 / 竞争环境 / 机会洞察 / 战略承接，则按一级目录依次输出，每个一级目录对应页面输出完成后暂停等待确认。

逐页稿必须像样板报告一样，按证据和结论逐步释放信息：

1. 先说明客户现状与问题；
2. 再分析行业环境；
3. 再分析消费者心智 / 需求；
4. 再分析竞品格局；
5. 再做企业自身扫描；
6. 再进行机会点交叉分析；
7. 最后进入战略承接和推进节奏。

页面应形成：问题铺垫 → 环境证据 → 消费者证据 → 竞品证据 → 自身证据 → 机会点 → 承接路径。不得在前两章提前输出机会点和大单品答案。

Agent 每次输出逐页内容前，必须先说明本批内容对应哪个客户可见一级目录，并输出“本章任务 / 本章不回答 / 本章预计页面”：

```markdown
# 本批生成内容：第一部分「项目概况」

本部分对应客户可见一级目录：
一、项目概况

## 本章任务
本章负责回答：
1. 本次项目背景是什么；
2. 客户为什么需要做这份建议书；
3. 本报告将如何展开。

## 本章不回答
本章暂不回答：
1. 后续机会点；
2. 大单品承接；
3. 战略推进动作。

## 本章预计页面
第 1 页：目录
第 2 页：项目背景
第 3 页：客户核心诉求
……
```

上述“本批生成内容 / 本部分对应客户可见一级目录 / 本章任务 / 本章不回答 / 本章预计页面”只属于对话中的生成过程控制信息，不是客户汇报内容。生成最终 Word 时必须删除，不得进入 Word 文档。

对话中的分批逐页稿属于“过程审核版”，可以保留页面类型、资料依据和章节任务。最终 Word / PPT 制作版必须删除页面类型、资料依据、页面正文标签、阶段判断标签和所有内部过程说明。

如果某个一级目录下页面数量过多，可在该一级目录内部拆分为多批。拆分原则：先按逻辑小节拆分，不按固定页数机械拆分；说明“仍属于同一个一级目录”；不得跨一级目录混合输出；不得把一个逻辑小节拆断；每批输出后暂停等待确认。

逐页内容生成阶段的确认单位不是页数，而是客户可见一级目录。Agent 应根据确认点 3 的目录和内部分析结构判断每部分需要多少页，使完整初稿落在 50-60 页区间。只有当所有客户可见一级目录及其内部分析结构完整生成后，才算完成建议书初稿。

#### 3.4.1.1 页间逻辑承接与客户视角

逐页内容不能只是一页一页孤立生成。每一页必须形成：

上一页结论
→ 本页进一步拆解 / 验证 / 推进
→ 下一页需要继续回答的新问题

每生成一页时，Agent 必须内部检查：

1. 上一页讲了什么；
2. 本页承接了什么；
3. 本页推进了什么；
4. 下一页为什么应该出现。

这些检查只用于内部生成，不得在最终 Word 中机械写出“承接上一页 / 引出下一页”等说明。

不得出现每页像独立笔记、上下页之间跳跃、前一页刚讲问题后一页突然给结论、没有铺垫就进入机会点、没有机会点确认就进入产品承接。

每页应是汇报故事线中的一个镜头，并形成：

问题出现
→ 问题被拆开
→ 外部环境验证
→ 竞品格局对照
→ 企业自身回看
→ 机会点浮现
→ 战略承接落地

最终 Word 不机械写“承接上一页 / 引出下一页”，只通过正文语言自然连贯。

最终 Word 是给客户汇报的页面脚本，不是内部分析日志。客户版汇报中，“客户”应替换为具体品牌名、企业名或“品牌自身”。

不得使用：

- 客户当前的问题是……
- 客户资料显示……
- 客户希望……
- 客户当前的核心诉求……
- 该客户……
- 本客户……

应改为：

- 御棠源当前的问题是……
- 从御棠源现有资料看……
- 御棠源希望解决的是……
- 当前品牌增长的核心矛盾是……
- 对御棠源而言……
- 本项目需要回答的是……

如果是其他品牌，自动替换为对应品牌名称、企业名称或“品牌自身”。

最终 Word 的正文要面向 PPT 制作，语言应更短、更直接、更有画面感。避免“资料显示 / 从现有资料看 / 本报告将 / 本建议书不会 / 阶段判断 / 后续需要 / 该页面说明 / 本章形成 / 当前资料中出现”等内部分析口吻。

最终 Word 页面正文不得大量重复“一句话 + 编号列表 + 阶段判断”的结构。同一章节内，连续页面不得都使用这种格式；连续 3 页出现类似结构时必须主动改写。阶段判断可作为内部思考，但不要在最终 Word 中机械写“阶段判断：”。重要判断可升级为页面标题，适合作为转折的判断可单独做成金句页、方向页或转折页。

### 3.4.2 逐页稿章节任务边界规则

逐页 PPT 内容脚本生成时，每个客户可见一级目录都有明确章节任务。Agent 不得跨章节提前输出后文结论。

如果客户可见一级目录类似：

```text
一、项目概况
二、问题诊断
三、环境分析
四、机会洞察
五、战略承接
```

则各部分默认任务为：

#### 一、项目概况

只回答：

- 本次项目背景是什么；
- 客户为什么需要做这份建议书；
- 客户提出了哪些表层诉求；
- 本项目的研究路径是什么；
- 本报告将如何展开。

不得输出：

- 已确认机会点；
- 战略大单品候选；
- 最终战略方向；
- 后续产品承接结论；
- 具体打法。

“项目概况”部分不得提前出现以下内容：

- 具体机会点名称；
- 战略大单品候选；
- 产品承接结论；
- “本建议书将以 XXX 为核心机会点”；
- “后续将优先判断 XXX 作为战略大单品”。

错误示例：“本建议书将以‘不垮气色’为核心机会点，兼顾‘术后元气恢复’的延展价值。”

正确表达：“本建议书将先从客户问题出发，依次完成行业、消费者、竞品和企业自身分析，最终推导具体机会点，并判断现有产品如何承接。”

#### 二、问题诊断

只回答：

- 客户表层问题是什么；
- 零散问题如何归因；
- 表面问题和根本问题是什么；
- 为什么这些问题本质上指向战略方向不清晰、增长主线不明确。

不得输出：

- 具体机会点；
- 具体产品承接；
- 最终战略方案。

#### 三、环境分析

只回答：

- 行业如何变化；
- 消费者需求 / 心智如何变化；
- 竞品格局是什么；
- 企业自身资产和问题是什么。

不得输出：

- 客户最终机会点；
- 客户应该占据什么；
- 大单品候选。

#### 四、机会洞察

只在本章正式输出：

- 四维交叉分析；
- 具体机会点；
- 机会点具体性检查；
- 最终机会点清单。

具体机会点只能在“机会洞察 / 行业潜在机会”章节正式输出。在此之前，可以说：

- 后续将通过四维交叉推导机会点；
- 当前分析将为机会点判断提供依据；
- 该结论可支撑后续机会方向。

但不得说：

- 本项目机会点是 XXX；
- 本建议书将以 XXX 为核心机会点；
- XXX 是战略大单品候选。

#### 五、战略承接

战略承接章节必须建立在机会点已经正式输出之后。只在机会点已经确认后输出：

- 哪个机会点作为主线；
- 现有产品池如何承接；
- 如果涉及大单品，哪个产品可作为候选；
- 如果现有产品无法承接，应如何打造；
- 后续推进节奏。

如果机会点尚未在前文章节正式提出，不得提前生成：

- 大单品候选；
- 产品承接判断；
- 渠道承接路径；
- 战略推进节奏。

### 3.5 用户极简使用方式与执行触发语

用户使用本 Skill 时不需要每次重复完整流程规则。详细规则由 Skill 文档、AGENTS.md 和部署提示词承担。Agent 必须主动读取并遵守 AGENTS.md、Skill 文档、部署提示词、客户资料文件夹、knowledge 相关资料。

用户只需要告诉 Agent：客户资料在哪里、从哪个阶段开始、当前确认点是否通过、是否继续下一步。

#### 新项目启动指令

```text
请按照本项目 Skill 执行。本次客户资料在 inputs/客户资料/客户名称/ 中，请从确认点 1 开始。
```

Agent 收到后应自动：读取规则文件、读取客户资料、读取相关知识库、执行确认点 1、在对话框完整展示确认点 1、暂停等待确认。

#### 标准阶段推进指令

1. “确认点 1 通过，继续下一步。” → 进入研究分析层（资料检索与用途归类 + 行业 / 消费者 / 竞对 / 企业自身分析）。
2. “研究依据库通过，继续确认点 2。” → 进入四维交叉机会点洞察。
3. “确认点 2 通过，继续确认点 3。” → 进入客户可见一级目录 + 内部分析结构。
4. “确认点 3 通过，开始生成第一部分。” → 进入逐页内容脚本生成，自动读取确认点 3 的客户可见一级目录，判断第一部分的章节任务，先输出本章任务 / 本章不回答 / 本章预计页面，再只生成第一部分应包含的内容，不提前输出后续章节结论，输出后暂停等待确认。
5. “第一部分通过，继续第二部分。” → 输出第二部分完整页面，并依次类推。

如果用户使用“继续生成第 11-20 页”这类页码指令，可以执行，但应优先提醒：“建议按一级目录分批输出，以保持建议书逻辑完整。”

#### 当前阶段自动识别规则

当用户输入“继续下一步”时，Agent 必须根据当前已完成阶段判断下一步：

- 确认点 1 刚通过 → 研究分析层（资料检索与用途归类）；
- 研究依据库刚通过 → 确认点 2；
- 确认点 2 刚通过 → 确认点 3；
- 确认点 3 刚通过 → 逐页内容脚本生成第一部分；
- 正在分批生成页面 → 继续生成下一部分。

“继续下一部分”按客户可见一级目录继续，而不是根据是否达到页数判断。“生成完整初稿”按目录逻辑完整生成，使初稿落在 50-60 页区间。如果 Agent 无法判断当前阶段，必须先询问，不得擅自跳步。

#### 用户短反馈修改规则

用户可用简短反馈要求修改（如“根本问题判断不准，重新判断”“确认点 2 机会点太虚，重新推导”“目录太散，重新整合”“竞对分析太浅，补充优势渠道与核心人群”）。Agent 收到反馈后，只修改对应阶段内容，不得擅自重跑全部流程。

## 5. 知识库结构

第一版 Skill 建议建立以下知识库。

### 5.1 机会点推导方法库

用于机会点洞察。机会点统一由四维交叉推导：

> **行业趋势 + 消费者需求 / 心智 + 竞品格局 + 企业自身优势 = 具体机会点**

四维含义：

1. 行业趋势：行业研究中得出的、指向某个方向的趋势（如品类升级方向、消费升级方向、渠道变化等）。
2. 消费者需求 / 心智：消费者关注点、痛点、消费场景、真实语言中体现的未被满足或被高度在意的需求。
3. 竞品格局：竞对分析得出的竞品共性与“未被充分呈现的方向”（注意：竞对阶段不下客户机会判断，只描述格局）。
4. 企业自身优势：企业自身扫描中“可支撑具体机会点的战略优势”。

机会点成立标准（四问全部能回答才算成立）：

1. 消费者为什么在意；
2. 竞品是否没有占住；
3. 企业自身是否能支撑；
4. 能否被产品或品牌表达承接。

机会点必须具体到消费者能感知、能表达、能传播的词或方向，参考样板报告中“软 / 0 添加”这类表达。任一问题无法回答，只能写“不成立”或“暂无法判断，需补充关键资料”。

不得在确认点、研究分析、机会洞察、目录生成、逐页建议书内容中，将‘三品破局 / 品牌破局 / 品类破局 / 产品破局 / 心智空位 / 工厂空位’作为判断框架、分析分类或客户可见输出表述使用。上述词语只允许出现在 Skill 规则的禁用说明中。

### 5.2 过往建议书案例库与样板报告逻辑库

用于参考：建议书结构、页面递进方式、表达节奏、逻辑推导方式、过渡页 / 反问页 / 金句页 / 思考页写法。

`knowledge/proposal_cases/` 下的样板报告逻辑文件，只用于学习建议书的制作逻辑、推导结构、维度框架与表达方法。不得复制其具体行业数据、客户内容、品牌结论、页面文案或图表内容。

### 5.3 行业研究规则库

行业研究采用漏斗式收窄：大行业 → 品类 → 细分赛道。必须包含：

1. 市场规模与增速
2. 行业发展阶段
3. 细分品类结构（行业全景 / 品类结构页）
4. 行业集中度
5. 行业消费趋势
6. 消费者痛点与诉求
7. 行业结论页（每段结论须说明可支撑的后续机会方向）

尽量呈现“国际 → 国内 → 细分”的递进。涉及市场规模、增速、份额、集中度、梯队等数据时，必须标注资料依据。若查不到当前年份数据，可使用最近可查年份替代并说明年份。

### 5.4 消费者需求 / 心智分析规则库

消费者需求 / 心智分析必须尽量输出：

1. 消费者关注点（尽量量化排序）
2. 消费者痛点（尽量量化排序）
3. 消费场景
4. 购买渠道 / 信息获取平台
5. 品牌认知（无提示 / 提示下，资料允许时）
6. 消费者真实语言（保留原话式标签，不得用抽象描述替代）

每类消费者结论都应能在机会点洞察阶段被点名引用。

### 5.5 竞对分析规则库

竞对分析的任务是看清竞争格局，不是直接给出客户机会点。

竞对分析通用必填只保留四个核心判断：

1. 品牌定位是什么（从口号、官网主图、品牌主张、核心传播判断它想占据什么位置，不只复述官方说法）；
2. 优势渠道是什么（主要销售渠道 / 内容渠道 / 线上线下触点）；
3. 核心人群是谁（结合产品、内容、渠道判断人群、场景、需求，不写泛泛标签）；
4. 优势产品是什么（代表产品 / 主推产品 / 优势单品及其核心卖点）。

生产实力、销售规模、品牌认知率、消费者感知特点、价格区间、保质期等，只作为“资料充分时的补充分析项”，不得作为通用必填项，更不得为凑维度而编造。

竞对分析结构（顺序固定）：

1. 确定核心竞品名单（默认 3-5 个，客户指定竞品优先）；
2. 逐个核心竞品拆解（覆盖四项核心判断；资料充分时补充补充项）；
3. 横向共性对比；
4. 竞争格局结论。

#### 核心竞品页数指引

核心竞品逐个拆解时，默认每个核心竞品用 1-2 页完成分析。每个核心竞品必须讲清四项核心判断：①品牌定位是什么；②优势渠道是什么；③核心人群是谁；④优势产品是什么。

- 如果某个竞品资料充分、战略动作复杂，或对本项目机会判断影响较大，可以适当增加页面。
- 如果资料不足，可以用一页完成，但必须标注“资料不足 / 需补充出处”。
- 不得机械规定每个竞品必须 4 页，也不得为了凑页数扩写竞品分析。
- 不得把生产实力、销售规模、品牌认知率、消费者感知特点作为通用必填项；这些只在资料充分时作为补充项。

竞对横向对比表不得设置“机会点”列。竞对横向表建议列为：

- 共性
- 差异
- 未充分呈现方向
- 阶段结论

竞对分析阶段不得直接输出客户机会点，不得输出“本客户机会空位 / 客户应如何破局 / 竞争空位初判 / 对本客户的启发”等结论。如需表达“未被充分覆盖的需求”，只能写成：“从竞品格局看，某些需求或表达尚未在主要竞品中被充分呈现。是否构成客户机会，需要在后续机会洞察中结合行业趋势、消费者需求与企业自身优势进一步判断。”

竞对分析是机会洞察的输入之一，不是机会洞察本身。客户机会点必须在机会点洞察阶段由四维交叉综合推导。

### 5.6 企业自身扫描规则库

完整企业自身扫描放在研究分析层（不放在确认点 1）。必须覆盖：

1. 现有业务
2. 产品基础
3. 研发能力
4. 渠道情况
5. 经营问题
6. 品牌问题
7. 机会与挑战

企业优势必须分为三类：

1. 仅企业端优势（消费者不易感知）；
2. 可支撑消费者选择理由的优势；
3. 可支撑具体机会点的战略优势。

自身优势应以“能否支撑某具体机会点”为标准分级；无法支撑任何机会的优势标注为“暂不构成战略优势”。不得使用“心智空位潜力”这类旧语言。

### 5.7 战略概念与品牌案例库

当建议书出现以下概念时，必须解释含义、商业意义，并用知名品牌案例具象化：定位、用户心智、差异化、价值主张、大单品、战略营销动作。

案例库结构建议：

```text
它给客户的启发：
适合在哪类建议书中引用：
```

### 5.8 页面表达与汇报节奏库

建议书页面类型包括：分析页、结论页、反问页、金句页、思考页、过渡页、转折页、关键判断页、概念解释页、案例说明页、目录页、数据页、对比页、表格页、路径页、总结页。

页面类型只作为 Agent 内部判断页面结构和汇报节奏的工具，不进入最终 Word / PPT 制作版。

页面表达规则：

- 分析页：标题写成结论句，正文用 1）2）3）有序证据支撑，先判断后证据。
- 数据页：一页一个数据结论，必须带数据来源，数据服务结论。
- 结论页：用结构化证据支撑一个结论，逻辑显性化。
- 表格页：表格必须输出结论。竞对横向对比表不得设“机会点”列（列为共性 / 差异 / 未充分呈现方向 / 阶段结论）；机会点交叉分析表才可设“机会点”列。
- 金句页：只保留一句话，不写多段解释。
- 反问页：只保留一个问句，不展开论证。

Agent 不得连续输出大量干巴巴分析页，必须根据逻辑节奏加入反问页、金句页、思考页或过渡页。

连续 4-5 页密集分析后，应考虑加入一页过渡页、金句页、反问页、方向页或总结页。从“问题诊断”进入“竞争环境”前应有承接页；从“竞争环境”进入“机会洞察”前应有转折页；从“机会洞察”进入“战略承接”前应有承接页。不要让整份建议书像分析笔记。

### 5.9 Word 版 PPT 页面脚本输出规则

所有最终内容必须按逐页 PPT 脚本输出，不得写成普通报告文章。整体遵循“标题即观点、正文即证据、数据标来源、表格出结论”。

生成最终 Word 前，必须做一次“PPT制作稿清洗”：

必须删除页面类型、页面正文标签、资料依据、资料依据标签、阶段判断标签、本批生成内容、本章任务、本章不回答、本章预计页面、暂停确认语、用户操作提示、质量检查报告、Agent 过程说明、Skill 执行过程说明和任何内部审核说明。

必须把“客户”替换为具体品牌名 / 企业名 / 品牌自身；把“客户资料显示”替换为“从现有资料看”或“从具体品牌现有资料看”；把“客户当前问题”替换为“具体品牌当前问题”；“客户版资料依据”等内部表达不得进入正文。

必须检查并优化：是否大量重复“一句话 + 编号列表 + 阶段判断”；是否有过多“资料显示 / 从现有资料看 / 本报告将 / 本建议书不会 / 后续需要 / 该页面说明 / 本章形成 / 当前资料中出现”等内部分析口吻；是否缺少过渡页、金句页、反问页、方向页或总结页；是否每页都能自然承上启下；是否更像 PPT 页面而不是研究报告。

## 6. RAG 调用方案

### 6.1 核心原则

1. 当前客户资料优先于过往案例。
2. 当前客户真实诉求优先于固定方法论。
3. 过往建议书案例与样板报告主要用于参考结构、节奏和表达，不得机械照搬，更不得复制其具体数据与结论。
4. 涉及市场数据、行业份额、竞品事实时，必须标注来源或注明需补充出处。
5. 如果资料不足或判断链条不完整，Agent 必须先追问，但只能停止一次。

### 6.2 调用节点

#### 节点 1：输入资料判断
输出：是否可以生成、缺失信息清单、最多 5 个追问问题。

#### 节点 2：客户诉求分析
输出：客户基本情况理解、客户明确提出的诉求与困惑、表面问题判断、深层原因 / 根本问题判断、是否属于品牌营销战略问题。

#### 节点 3：资料检索与用途归类
调用：行业研究规则库、竞对分析规则库、消费者需求规则库、外部联网检索、客户资料、内部知识库。
输出：研究依据库。

研究依据库必须包含：行业资料及来源、消费者需求 / 心智资料及来源、竞对资料及来源、企业自身资料归类、资料可靠性判断、对建议书页面的支撑关系。每条资料保留资料内容、来源名称、机构 / 发布方、年份、链接或文件名、可靠性等级（可直接引用 / 谨慎引用 / 仅作方向参考）、可支撑的建议书页面。

MEMORY.md、agent_memory 或任何项目记忆文件不得进入研究依据库，不得作为行业、消费者、竞对、客户或建议书资料来源；只能用于流程衔接和阶段判断。

#### 节点 4：研究分析生成
输出：行业研究、消费者需求 / 心智分析、竞对分析、企业自身扫描。

要求：
- 行业研究漏斗式收窄，含行业全景 / 品类结构页，数据标来源，每段结论指向机会方向。
- 消费者需求 / 心智分析输出关注点、痛点、场景、渠道 / 平台、认知、真实语言。
- 竞对分析逐个竞品拆解（四项核心判断）→ 横向共性对比 → 竞争格局结论；不输出客户机会点。
- 企业自身扫描覆盖七项内容，自身优势分三类。

#### 节点 5：机会点洞察
调用：研究依据库、行业研究结果、消费者需求 / 心智分析结果、竞对分析结果、企业自身扫描结果、品牌案例库。
输出：确认点 2：具体机会点洞察（含机会点交叉分析表）。

机会点统一由四维交叉推导，必须具体到可感知词，并通过四问检验。判断结论只能用：成立 / 不成立 / 暂无法判断，需补充关键资料。

#### 节点 6：建议书目录结构生成
输出：确认点 3：客户可见一级目录、目录命名逻辑说明、内部分析结构、页面标题方向提示、待确认事项。

#### 节点 7：概念解释与案例具象化
当建议书出现定位、心智、差异化、价值主张、大单品等概念时，调用战略概念与品牌案例库，输出概念解释页、案例说明页。

#### 节点 8：页面节奏与 Word 输出
输出：逐页 PPT 内容脚本（50-60 页区间）。

## 7. 客户需求理解模块

### 7.1 客户需求类型

Agent 应判断客户属于以下哪类：

1. 明确需求型：客户清楚要做营销战略、定位、品牌战略。
2. 模糊增长型：客户只说增长遇到瓶颈、想增长。
3. 核心命题型：客户有明确战略方向（如某产品高端化）但不知道怎么做。
4. 碎片问题型：客户提出人群、包装、品牌故事、传播等零散问题。

### 7.2 客户需求理解规则

- Agent 不得直接复述客户原话作为最终判断。
- 客户说“想做营销战略”时，必须诊断真实问题，而不是照单全收。
- 客户说“想增长”但说不清问题时，确认点 1 只能围绕客户明确提出的诉求和现有资料做谨慎判断；如需进一步确认，应标注“需补充确认”。
- 客户提出碎片问题时，可以归纳成上位战略议题，但只能基于客户明确提出的信息，不得把资料中出现的所有产品、渠道、传播、人群、包装、卖点都主动写成客户诉求。
- 用户或客户明确提出的具体问题、执行问题，不得直接等同于项目根本课题。
- 如果判断项目属于品牌营销战略问题，深层原因 / 根本问题通常应上收到同一个核心问题：战略方向不清晰，导致品牌增长主线不明确，产品、渠道、传播、资源和消费者选择理由无法形成合力；但必须结合客户实际资料自然表达，不能机械套话。
- 企业资料在确认点 1 中只用于辅助理解客户现状和支撑根本问题判断，不用于主动制造新的客户诉求。
- 如果 Agent 从资料中发现潜在问题，只能在“深层原因 / 根本问题判断”中谨慎提及，不得作为“客户明确提出的诉求与困惑”输出。

### 7.3 明确诉求转译示例

| 客户表层表达 | 归纳后的战略议题 |
|---|---|
| 包装不好看、VI 不好看、色彩不突出 | 视觉战略问题 |
| 品牌故事不会讲 | 品牌价值表达问题 |
| 营销做了但没效果 | 传播没有围绕核心战略展开 |
| 不知道核心优势怎么宣传 | 差异化没有被转译成价值主张 |
| 人群感觉不对 | 目标人群与品牌价值未匹配 |
| 产品卖点很多但说不清 | 缺少核心选择理由 |

以上示例只适用于客户或用户已经明确提出对应问题的情况。确认点 1 不主动从客户资料中发散归纳大量问题；如果客户没有明确提出某类问题，Agent 不得把该类问题写成客户诉求。

当客户明确提出以下问题时，Agent 不得直接把它们当成项目课题：想选大单品；产品太多，不知道主推哪个；包装不好看；VI 不突出；品牌故事不会讲；传播没有效果；不知道核心优势怎么宣传；人群感觉不对；渠道增长乏力；卖点很多但说不清；品牌认知弱。

这些明确提出的问题在品牌营销战略项目中，可被归纳为战略方向不清晰的不同表现。大单品、包装、传播、人群、渠道、卖点等，可以作为后续战略承接问题，但不能在确认点 1 被直接定义为根本课题。

## 8. 项目问题定义模块

### 8.1 问题拆解方式

确认点 1 的问题定义必须围绕客户明确提出的诉求与困惑展开，默认拆成：

1. 客户基本情况理解（客户是谁、所属行业 / 品类、主营产品 / 服务、当前基础资产、当前大致发展阶段）
2. 客户明确提出的诉求与困惑（只整理用户或客户明确告诉 Agent 的诉求、目标、困惑和问题）
3. 表面问题判断（基于客户明确提出的信息判断表面上看是什么问题）
4. 深层原因 / 根本问题判断（结合客户资料和企业基本情况，分析表面问题背后的深层原因）
5. 是否属于品牌营销战略问题（判断本次项目是否适合调用本 Skill）

确认点 1 的核心不是“尽可能多地发现问题”，而是基于客户明确提出的诉求与困惑，判断本次项目真正要解决的核心问题。不得把 Agent 自己从资料里推测出来的问题，混入“客户明确提出的诉求与困惑”。如果资料中没有明确诉求，应写：“当前客户明确诉求不足，需补充确认。”

示例：

```text
客户表面问题：传播没有抓手，不知道怎么宣传。
根本问题：不是传播动作不足，而是品牌没有形成清晰差异化，也没有建立稳定的核心表达。
```

如果项目属于品牌营销战略问题，根本问题不得停留在某个具体执行动作上。

错误示例：

- 根本问题是如何选择大单品；
- 根本问题是如何优化包装；
- 根本问题是如何提升传播效率；
- 根本问题是如何做产品结构梳理。

正确表达应类似：

- 根本问题不是单点执行问题，而是品牌战略方向不清晰；
- 客户缺少一个能够统领产品、渠道、传播和消费者认知的增长主线；
- 企业优势、消费者需求、竞品格局和产品承接之间尚未形成清晰闭环；
- 因此，后续建议书需要先找到战略机会点，再判断产品、渠道和传播如何承接。

判断根本问题时，可以初步提及客户已有资产与问题（如过度依赖某单一业务、缺乏能打的自有品牌），但只能作为根本问题判断的佐证，不得在确认点 1 展开完整企业自身扫描。

如果 Agent 从资料中发现潜在问题，只能在“深层原因 / 根本问题判断”中谨慎提及，不得作为“客户明确提出的诉求与困惑”输出。

### 8.2 表达风格

问题定义部分允许尖锐，但必须有依据、有推理，不能武断。建议使用：

- 不是 A，而是 B
- 表面上看是 A，本质上是 B
- 客户以为要解决 A，但真正影响增长的是 B

## 9. 研究分析、机会点洞察与目录生成模块

### 9.1 生成顺序

1. 确认点 1 通过后，先进行资料检索与用途归类，形成研究依据库。
2. 基于研究依据库，完成行业研究、消费者需求 / 心智分析、竞对分析、企业自身扫描。
3. 基于四维交叉，推导具体机会点。
4. 输出确认点 2：具体机会点洞察。
5. 在确认点 2 暂停，等待用户确认。
6. 确认点 2 通过后，生成客户可见一级目录 + 内部分析结构。
7. 在确认点 3 暂停，等待用户确认。
8. 确认点 3 通过后，按客户可见一级目录分批生成逐页内容脚本；每完成一个一级目录对应内容后暂停确认。

确认点 1 通过后不得立即生成目录，必须先完成研究分析层并推导出具体机会点，然后才能输出确认点 2。

### 9.2 资料检索与用途归类节点

资料检索与用途归类节点必须放在确认点 1 之后、确认点 2 之前。它不是正文生成，而是机会点洞察的前置基础。

硬性规则：涉及市场规模、增速、份额、行业排名、消费者洞察、竞品事实时必须有资料依据；查不到最新年份可用最近可查年份并说明；不得编造数据；没有依据的判断只能写成“待验证假设”或标注“需人工确认 / 需补充出处”。

### 9.3 确认点 2：具体机会点洞察

确认点 2 不再输出三品破局判断。它的核心任务是：基于行业趋势、消费者需求 / 心智、竞品格局、企业自身优势，推导具体机会点。

确认点 2 必须在完成行业研究、消费者需求 / 心智分析、竞对分析、企业自身扫描后输出，并必须包含“机会点交叉分析表”和“机会点具体性检查”。

机会点交叉表建议列为：

- 行业趋势
- 消费者需求 / 心智
- 竞品格局
- 企业自身优势
- 机会点

机会点不是一句战略方向，而是由四维证据压缩出来的具体机会词：

行业趋势 + 消费者需求 / 心智 + 竞品格局 + 企业自身优势 = 具体机会点。

机会点必须具体到消费者能感知、能表达、能传播的词或方向。合格机会点应像“软 / 0 添加 / 不干 / 无糖 / 轻负担 / 即饮 / 便携 / 高蛋白 / 低 GI / 好吸收”一样具体；这些只是表达颗粒度示例，不得机械套用。不得停留在“健康、专业、有效、可信、功能化、日常化”等泛方向。

每个机会点必须回答四个问题：

1. 消费者为什么在意：是否是痛点、关注点、购买理由、使用场景，或真实语言中反复出现的词。
2. 竞品是否没有占住：主要竞品是否没有明确表达；是否只是行业共性；是否已被某个竞品强占；如果已被强占，则不能作为客户机会点。
3. 企业自身是否能支撑：是否有产品、技术、供应链、原料、渠道、品牌资产支撑；如果只是企业愿望，没有能力支撑，则不成立。
4. 能否被品牌表达或产品承接：是否能变成消费者听得懂的表达；是否能被现有产品承接；如果不能承接，是否需要打造新产品。

四问不能回答清楚，则不得写成成立的机会点。

确认点 2 标准输出格式：

```markdown
# 确认点 2：具体机会点洞察

## 一、本轮研究依据概览
说明本轮调用了哪些资料：客户资料、行业资料、消费者需求 / 心智资料、竞对资料、过往案例库。

## 二、行业与消费者需求 / 心智关键判断
- 输出行业层面关键判断（漏斗式收窄到细分赛道）；
- 输出消费者关注点、痛点、场景、渠道、认知、真实语言关键判断；
- 每个判断必须有资料依据；没有依据的标注“需人工确认”或“需补充出处”。

## 三、竞品格局关键判断
- 输出主要竞品的品牌定位、优势渠道、核心人群、优势产品；
- 输出竞品共性、差异与未充分呈现的方向；
- 仅描述竞争格局，不在此输出客户机会点；
- 逐页内容中的“竞争环境”章节须展开为核心竞品逐个分析，不得用本摘要替代。

## 四、企业自身优势判断
- 现有业务、产品基础、研发能力、渠道、经营问题、品牌问题、机会与挑战概览；
- 自身优势三类划分：仅企业端优势 / 可支撑消费者选择理由的优势 / 可支撑具体机会点的战略优势。

## 五、机会点交叉分析表
用表格逐行推导每个候选机会点：

| 行业趋势 | 消费者需求 / 心智 | 竞品格局 | 企业自身优势 | 机会点 |
|---|---|---|---|---|
| …… | …… | …… | …… | （具体到可感知词，如“软 / 0 添加”式表达） |

## 六、机会点具体性检查
对每个候选机会点逐一检查：

- 机会点名称：
- 是否是消费者能感知的具体词：
- 消费者为什么在意：
- 竞品是否没有占住：
- 企业自身是否能支撑：
- 是否能被品牌表达或产品承接：
- 判断结论：成立 / 不成立 / 暂无法判断
- 如果不够具体，需要继续压缩成什么表达：

只有通过“机会点具体性检查”的，才能进入最终机会点清单。

## 七、最终机会点清单
最终机会点清单不得超过 2-3 个。每个机会点必须是具体词或具体方向，不得输出一堆泛泛战略方向。

### 机会点一：XXX
- 机会点表达：
- 行业趋势证据：
- 消费者需求 / 心智证据：
- 竞品格局证据：
- 企业自身优势证据：
- 为什么成立：
- 可承接方向：

### 机会点二：XXX
同上。

机会点必须具体，不得用“做差异化 / 做健康 / 做高端 / 提升品牌力 / 有效食养 / 功能场景 / 可信赖 / 专业化 / 日常化功能食品 / 高感知功能场景 / 制药级药食同源 / 有验证依据 / 安全有效 / 长期调理”等泛泛表达替代。上述词可以作为背景、趋势、定位素材或机会方向的上位描述，但不是最终机会点。如果出现这类泛词，必须继续追问：它到底具体体现为消费者能感知的哪个词？它解决的是哪个具体痛点？它能被哪个产品或表达承接？四问任一无法回答，只能写“不成立”或“暂无法判断，需补充关键资料”。

## 八、战略大单品分析（仅在客户明确提出大单品需求时输出）
见 9.3.1。

## 九、等待用户确认
请用户确认：
1. 机会点是否具体、是否成立；
2. 四维交叉推导是否有说服力；
3. 哪些判断需要加强或调整；
4. 是否进入确认点 3：客户可见一级目录 + 内部分析结构。
```

#### 9.3.1 战略大单品分析（仅在客户明确提出大单品需求时输出）

大单品是已确认战略机会点的承接物，不是机会点本身。只在客户明确提出大单品需求时启动。

战略大单品分析只保留三项：

### 1. 战略机会点
- 已确认的战略机会点：
- 机会点成立原因：
- 该机会点需要什么样的产品承接：

### 2. 现有产品池
- 现有产品 / 产品系列：
- 各产品对应的功能、场景、人群或需求：
- 初步具备承接可能的产品：

### 3. 与机会点匹配度
- 匹配度判断标准：该产品能否兑现机会点要求的消费者可感知差异化；
- 与机会点匹配度最高的产品及匹配理由：
- 不匹配或弱匹配的产品：
- 判断结论：
  - 现有产品可以承接 → 建议作为战略大单品候选；
  - 现有产品无法承接 → 不得硬选，输出“围绕机会点打造战略大单品”的建议；
- 需要人工确认的信息：

复购率、毛利、渠道、传播潜力等只能作为“与机会点匹配度”的辅助判断依据，不得作为固定目录强制展开，也不得写成复杂产品评分模型。

### 9.4 确认点 3：建议书目录结构确认

确认点 3 的任务是：在确认点 2 的机会点已确认后，反推整份建议书的汇报结构。目录不是研究开始前的框架假设，而是研究完成后的汇报结构。

- 不能在完成研究分析层和机会点判断之前生成正式目录。
- 目录必须围绕已确认的机会点组织。
- 一级目录是汇报结构，不是研究清单。
- 内部分析结构是生成页面的工作结构，不是客户可见目录。

#### 9.4.1 客户可见一级目录定义

客户可见一级目录必须具备：概括性、逻辑性、简洁性、整合性、客户可读性。一般 3-5 个，适合直接放入 PPT 目录页。不得把具体机会点、具体产品名、具体战略结论直接写成客户可见一级目录。

#### 9.4.2 标题层级规则

1. 客户可见一级目录：用于 PPT 目录页，高度概括。示例：项目概况 / 问题诊断 / 竞争环境 / 机会洞察 / 战略解法；或 发现问题 / 看清环境 / 判断机会 / 明确方向 / 落地承接。
2. 内部分析结构：给 Agent 后续生成页面使用，可以更具体，如“客户表层诉求与根本问题”“行业全景与细分赛道”“竞对竞争格局”“机会点交叉分析”“战略大单品承接”。
3. 页面标题 / 章节结论标题：用于具体页面，可表达具体判断、观点、金句或结论。

#### 9.4.3 客户可见一级目录禁止写法

1. 不得写成过长的判断句；
2. 不得直接写具体产品名；
3. 不得直接写具体机会点；
4. 不得把战略结论提前写成一级目录；
5. 不得把每个分析模块平铺成一级目录；
6. 不得把页面标题当作一级目录；
7. 不得为了追求观点感而牺牲概括性和结构感。

#### 9.4.4 客户可见一级目录推荐结构库

以下是目录组织方式参考，不得机械套用，标题必须结合客户改写：

输出确认点 3 前必须自检：

## 10. 章节内容生成规则

### 10.0 逐页 PPT 页面表达与信息释放总则

逐页稿不是报告段落拆页，而是可直接转成 PPT 的页面脚本。每页正文必须短、结构化、适合上屏。

页面表达规则：

- 分析页：一句观点判断；2-4 条结构化证据；必要时一行阶段判断。
- 数据页：一句数据结论；一个数据表 / 数据点说明；一句“这说明什么”。
- 结论页：一个核心结论；2-3 条支撑依据；不写成长段论述。
- 金句页：只保留一句话；不写解释。
- 反问页：只保留一个问句；不写解释。
- 表格页：必须输出结论；竞对横向对比表不得设置“机会点”列，机会点交叉分析表才可设置“机会点”列。

禁止事项：

- 不得在一页中写 5 段以上正文；
- 不得用报告段落代替 PPT 页面脚本；
- 不得把一页写成小作文；
- 不得在金句页 / 反问页写多句解释。

信息释放必须遵循：问题铺垫 → 环境证据 → 消费者证据 → 竞品证据 → 自身证据 → 机会点 → 承接路径。前文可为后文机会判断埋证据，但不得提前宣布后文机会点或战略承接答案。

### 10.1 客户诉求与问题界定

至少包含：品牌简介页、客户诉求整理页、表面问题与根本问题页、项目核心命题页、反问页 / 金句页 / 思考页。

### 10.2 企业自身扫描（在研究分析层）

完整企业自身扫描覆盖现有业务、产品基础、研发能力、渠道情况、经营问题、品牌问题、机会与挑战。自身优势分三类：仅企业端优势、可支撑消费者选择理由的优势、可支撑具体机会点的战略优势。

### 10.3 行业研究

漏斗式收窄：大行业 → 品类 → 细分赛道。必须包含市场规模与增速、行业发展阶段、细分品类结构（行业全景 / 品类结构页）、行业集中度、行业消费趋势、消费者痛点与诉求、行业结论页。尽量呈现国际 → 国内 → 细分递进。每段行业结论须说明可支撑的后续机会方向，数据须标来源。

### 10.4 竞对分析

当生成客户可见一级目录中的“竞争环境 / 竞争环境分析 / 看清环境”或类似章节时，必须按以下结构展开：

1. 行业与消费者环境；
2. 核心竞品逐个拆解（每个竞品覆盖品牌定位、优势渠道、核心人群、优势产品四项核心判断；资料充分时补充生产实力、销售规模、品牌认知率、消费者感知特点等补充项）；
3. 竞品横向共性对比（表格列为：共性 / 差异 / 未充分呈现方向 / 阶段结论，不得设“机会点”列）；
4. 竞争格局结论。

默认选择 3-5 个核心竞品，客户指定竞品优先。竞品很多时分为核心竞品（逐个拆解）与参考竞品（表格化补充）。竞对阶段不输出客户机会点、不输出“对本客户的启发 / 机会空位 / 竞争空位初判”。

核心竞品页数指引：默认每个核心竞品用 1-2 页完成，讲清品牌定位、优势渠道、核心人群、优势产品四项核心判断；资料充分、战略动作复杂或对机会判断影响较大时可适当增加页面；资料不足可用一页完成并标注“资料不足 / 需补充出处”。不得机械规定每个竞品必须 4 页，不得为凑页数扩写竞品分析。

如需表达“未被充分覆盖的需求”，只能写成：“从竞品格局看，某些需求或表达尚未在主要竞品中被充分呈现。是否构成客户机会，需要在后续机会洞察中结合行业趋势、消费者需求与企业自身优势进一步判断。”

### 10.5 机会点洞察

机会洞察板块进入前，先用反问页 / 思考页 / 金句页承接前文，引出“如何破局 / 机会在哪”的问题（反问页只保留一个问句）。

机会点统一由四维交叉推导：

> 行业趋势 + 消费者需求 / 心智 + 竞品格局 + 企业自身优势 = 具体机会点

机会洞察阶段必须回答：

1. 行业趋势是否提供机会方向；
2. 消费者需求 / 心智中是否存在被高度在意或未满足的点；
3. 竞品格局中哪些方向尚未被充分呈现；
4. 企业自身是否有可支撑具体机会点的战略优势；
5. 由四维交叉能得出哪些具体机会点；
6. 每个机会点是否通过四问检验（消费者为何在意 / 竞品是否没占住 / 自身能否支撑 / 能否被承接）。

机会点必须具体到消费者能感知、能表达、能传播的词或方向，参考样板报告“软 / 0 添加”式表达颗粒度，但不得机械套用样板词。不得用“做差异化 / 做健康 / 做高端 / 提升品牌力 / 有效食养 / 功能场景 / 可信赖 / 专业化 / 日常化功能食品 / 高感知功能场景 / 制药级药食同源 / 有验证依据 / 安全有效 / 长期调理”等泛泛表达充当机会点。判断结论统一为：成立 / 不成立 / 暂无法判断，需补充关键资料。

确认点 2 和机会洞察正文均必须进行“机会点具体性检查”：逐一判断机会点是否是消费者能感知的具体词、消费者为什么在意、竞品是否没有占住、企业自身是否能支撑、是否能被品牌表达或产品承接。最终机会点清单不得超过 2-3 个。

机会点交叉分析表才可设“机会点”列。

### 10.6 战略大单品分析（仅在客户明确提出大单品需求时）

只有当客户明确提出大单品相关需求时，才单独启动“战略大单品分析”模块。如果客户没有大单品相关诉求，Agent 不得默认加入。

判断顺序：先完成研究分析层与机会点洞察 → 明确具体机会点 → 回到客户现有产品池判断是否有产品能承接该机会点 → 能承接的现有产品才可定义为战略大单品候选；不能承接则提出“围绕机会点打造战略大单品”的建议。

战略大单品分析只围绕三项：战略机会点、现有产品池、与机会点匹配度（匹配度=该产品能否兑现机会点要求的消费者可感知差异化）。复购率、毛利、渠道、传播潜力等只作匹配度辅助依据，不得作为固定目录，不得写成复杂产品评分模型。

### 10.7 战略推进节奏（落地承接）

如需输出落地承接，应把机会点转化为分阶段推进节奏：结合企业现实资源，区分基础业务、拓展业务和品牌打造，体现“先夯实基础，再逐步打造品牌”，避免战略悬空。

## 11. 人机确认节点

本 Skill 设置 3 个强制确认点。

### 确认点 1：客户需求理解 + 项目问题定义后

Agent 必须在对话框中完整展示以下五项，然后暂停等待确认：

1. 客户基本情况理解
2. 客户明确提出的诉求与困惑
3. 表面问题判断
4. 深层原因 / 根本问题判断
5. 是否属于品牌营销战略问题

确认点 1 的核心不是“尽可能多地发现问题”，而是基于客户明确提出的诉求与困惑，判断本次项目真正要解决的核心问题。企业资料在确认点 1 中只用于辅助理解客户现状和支撑根本问题判断，不用于主动制造新的客户诉求。可以初步提及客户已有资产与问题，但只能作为根本问题判断的佐证，不得在确认点 1 展开完整企业自身扫描。确认点 1 不再判断“定位是否清晰 / 差异化是心智空位还是工厂空位 / 是否调用三品破局模型”。不得只保存文件而不展示确认点内容。

确认点 1 必须按以下格式输出：

```markdown
# 确认点 1：客户需求理解与项目问题定义

## 1. 客户基本情况理解
- 客户是谁；
- 所属行业 / 品类；
- 主营产品 / 服务；
- 当前具备的基础资产；
- 当前大致发展阶段。

要求：只做简要理解，不展开研究分析，不提前下战略结论。

## 2. 客户明确提出的诉求与困惑
只整理用户或客户明确告诉 Agent 的诉求、目标、困惑和问题。

如果资料中没有明确诉求，应写：
“当前客户明确诉求不足，需补充确认。”

## 3. 表面问题判断
基于客户明确提出的诉求与困惑，判断这些问题表面上看是什么。

注意：表面问题必须来自客户明确提出的信息，不要主动扩展成很多零散问题。

## 4. 深层原因 / 根本问题判断
结合客户资料和企业基本情况，分析这些表面问题背后的深层原因。

如果该项目属于品牌营销战略问题，根本问题通常应上收到：
战略方向不清晰，导致品牌增长主线不明确，产品、渠道、传播、资源和消费者选择理由无法形成合力。

但必须结合客户实际资料，用自然语言表达，不能机械套话。

## 5. 是否属于品牌营销战略问题
判断本次项目是否适合调用本 Skill。

如果属于：说明为什么属于品牌营销战略问题。
如果不属于：说明更适合什么类型的方案，例如招商方案、渠道方案、传播执行方案、电商运营方案等。
```

确认点 1 输出后必须暂停等待用户确认，不得进入研究分析层。

### 确认点 2：具体机会点洞察

Agent 必须在完成资料检索与用途归类、行业研究、消费者需求 / 心智分析、竞对分析、企业自身扫描后，在对话框完整展示确认点 2，然后暂停等待确认。

确认点 2 必须输出：本轮研究依据概览、行业与消费者需求 / 心智关键判断、竞品格局关键判断、企业自身优势判断、机会点交叉分析表、机会点具体性检查、最终机会点清单、战略大单品分析（仅客户提出时）、等待确认的问题。

机会点必须具体、由四维交叉证据压缩出来、通过四问检验和机会点具体性检查。判断结论统一为：成立 / 不成立 / 暂无法判断，需补充关键资料。不得只保存文件而不展示确认点内容。

### 确认点 3：建议书目录结构确认

Agent 必须在确认点 2 的机会点确认后，反推整份建议书的汇报结构，并在对话框完整展示确认点 3。

确认点 3 必须输出：客户可见一级目录（3-5 个，高度概括，不得写成具体机会点 / 产品名 / 页面标题）、目录命名逻辑说明、内部分析结构、页面标题方向提示、等待确认的问题。不得只保存文件而不展示确认点内容。

## 12. 质量检查模块

Agent 生成完整内容后，必须在对话框单独输出《建议书质量检查报告》。不打分。质检只看 3 个维度。

质量检查报告是内部检查结果，不是客户汇报页面内容。它只能在对话中单独输出，或另存为独立 Markdown 文件（例如 `outputs/质量检查报告.md`），不得写入最终 Word 文档末尾。最终 Word 的最后一页应是客户可见的总结页、下一步建议页或战略推进页。

### 12.1 逻辑链条是否完整通顺

- 客户诉求到问题定义是否自然；
- 问题定义到研究分析层是否自然；
- 行业、竞对、消费者、企业自身是否支撑机会点；
- 机会点是否由四维交叉得出并通过四问检验；
- 机会点是否具体（非泛泛表达）。

### 12.2 客户需求理解整理是否精准

- 是否整理客户表层诉求；
- 是否提炼客户零散问题；
- 是否归纳为上位战略议题；
- 是否拆解表面问题和根本问题；
- 是否准确判断是否属于品牌营销战略问题。

### 12.3 表达节奏是否合理

- 是否只有连续分析页，缺少结论页；
- 是否在关键转折处加入反问页（一个问句）、思考页或金句页（一句话）；
- 是否在讲战略概念时做了概念解释；
- 是否通过案例帮助客户理解；
- 是否避免过度学术化表达；
- 总页数是否落在 50-60 页区间。

### 12.4 修改规则

Agent 不自动重写。用户指出问题后，Agent 根据用户意见修改一次，优先保持原建议书主线。用户用简短反馈要求修改时，只修改对应阶段或指定范围，不得擅自重跑全部流程。

## 13. 禁止事项

1. 不得编造行业数据、市场规模、市场份额、增长率、行业排名。
2. 不得直接套模板；过往案例与样板报告只学逻辑，不得复制其具体数据、客户内容、品牌结论、页面文案或图表。
3. 不得跳过确认点。
4. 不得在资料不足时反复停止；最多停止一次。
5. 不得把客户原话直接当作最终问题定义。
6. 不得在确认点 1 展开完整企业自身扫描；确认点 1 默认按五项轻量结构输出；不得沿用旧版的零散问题整理项或归因小节；不得主动从客户资料中发散归纳产品、渠道、传播、人群、包装、卖点等大量问题；不得把资料中出现的所有信息都归纳为客户问题；不得把客户明确提出的具体问题或执行问题直接等同于项目根本课题。
7. 不得在确认点、研究分析、机会洞察、目录生成、逐页建议书内容中，将‘三品破局 / 品牌破局 / 品类破局 / 产品破局 / 心智空位 / 工厂空位’作为判断框架、分析分类或客户可见输出表述使用。上述词语只允许出现在 Skill 规则的禁用说明中。
8. 不得在确认点 1 使用上述旧框架判断“定位是否清晰 / 差异化类别 / 是否调用旧模型”。
9. 不得跳过资料检索与用途归类节点，直接生成行业、消费者或竞对相关页面。
10. 不得在确认点 1 通过后立即生成目录。
11. 不得在机会点未确认前生成正式目录。
12. 不得在行业研究中只堆数据而不指向机会方向（每段行业结论须说明可支撑的机会方向）。
13. 不得省略消费者关注点、痛点、场景、渠道与真实语言；不得用抽象描述替代消费者真实语言。
14. 不得把竞对分析的补充项（生产实力、销售规模、品牌认知率、消费者感知特点等）当作通用必填项强行填写或编造。

17. 不得只凭竞品分析一个维度判断客户机会点；机会点必须由四维交叉综合推导。
18. 不得把泛泛表达写成机会点（做差异化、做健康、做高端、提升品牌力、有效食养、功能场景、可信赖、专业化、日常化功能食品、高感知功能场景、制药级药食同源、有验证依据、安全有效、长期调理等）。这些词可以作为背景、趋势、定位素材或机会方向的上位描述，但不是最终机会点；如出现这类泛词，必须继续追问其对应的消费者可感知具体词、具体痛点和产品或表达承接方式。
19. 不得输出未回答四问、未通过机会点具体性检查却写成“成立”的机会点。
20. 不得用“具备验证价值 / 可能有机会 / 有一定空间”等模糊表达替代明确机会判断。
21. 战略大单品分析只在客户明确提出大单品需求时启动；大单品是机会点的承接物，不是机会点本身。
22. 不得让大单品脱离已确认机会点独立选择；现有产品无法承接机会点时不得硬选，应提出围绕机会点打造战略大单品的建议。
23. 不得把复购率、毛利、渠道、传播潜力等写成大单品固定目录；这些只能作为匹配度辅助依据。

25. 不得把分析模块平铺成客户可见一级目录。
26. 不得把客户可见一级目录写成页面标题、具体战略结论或具体产品名 / 机会点。

31. 数据页不得省略数据来源。
32. 表格页必须输出结论。

36. 不得为控制页数而压缩行业研究、竞对分析、机会洞察或大单品分析，也不得无限展开。
37. 不生成公司介绍、服务流程、服务板块、报价、团队介绍、项目周期、交付成果。

49. 不得把 MEMORY.md、agent_memory 或任何项目记忆文件作为行业、消费者、竞对、客户或建议书资料来源。

## 15. 总体执行流程

```text

↓
资料完整度判断
↓
如严重不足：停止一次，最多追问 5 个问题
↓
客户需求理解
↓
项目问题定义
↓
【确认点 1：需求理解 + 问题定义（五项轻量输出）】
↓
研究分析层：资料检索与用途归类 → 行业研究、消费者需求 / 心智分析、竞对分析、企业自身扫描
↓
四维交叉推导具体机会点（行业趋势 + 消费者需求 / 心智 + 竞品格局 + 企业自身优势 = 具体机会点）
↓
【确认点 2：具体机会点洞察（含机会点交叉分析表）】
↓
生成客户可见一级目录 + 内部分析结构
↓
【确认点 3：目录结构确认】
↓
```
