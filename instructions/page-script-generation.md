# 逐页 PPT 内容脚本生成执行说明

## 1. 本阶段目的

本阶段的目的，是在确认点 3 已通过后，根据客户可见一级目录和内部分析结构，逐部分生成 Word 版 PPT 页面内容脚本。

本阶段不是重新做研究，不是重新推导机会点，也不是自由发挥写报告。

最终 Word 文档不是过程记录，也不是 Agent 执行日志，而是给 PPT 制作方使用的页面脚本。最终 Word / PPT 制作版中每页只保留页码、页面标题和页面正文内容，不展示页面类型、页面正文标签、资料依据、资料依据标签或阶段判断标签。

## 2. 启动条件

只有在以下条件满足后，才能进入逐页生成阶段：

1. 确认点 1 已通过；
2. 研究分析层已完成；
3. 确认点 2 已通过；
4. 确认点 3 已通过；
5. 客户可见一级目录已确认；
6. 内部分析结构已确认。

如果确认点 3 未通过，不得直接生成逐页脚本。

## 3. 分批输出规则

逐页脚本必须按客户可见一级目录分批输出。

不得默认每 10 页输出一批。

每一批开始前必须先在对话中输出以下过程控制信息，用于用户审核：

```markdown
本批生成内容：第 X 部分「章节名」

本章任务：
1. ……
2. ……
3. ……

本章不回答：
1. ……
2. ……
3. ……

本章预计页面：
第 X 页：……
第 X+1 页：……
```

然后再开始逐页生成。

以上内容属于“生成过程控制信息”，不是汇报内容，不得写入最终 Word 文档。

每一批结束后必须暂停，等待用户确认。

用户确认后，才能继续下一部分。

## 4. 页面标准格式

过程审核版每一页必须使用以下格式：

```markdown
第 X 页：页面标题

页面类型：
目录页 / 章节页 / 结论页 / 分析页 / 数据页 / 对比页 / 表格页 / 金句页 / 反问页 / 路径页 / 总结页

页面正文：
……

资料依据：
……
```

过程审核版不得缺少页码、页面标题、页面类型、页面正文、资料依据，便于用户在对话中审核。

最终 Word / PPT 制作版每一页只保留：

```markdown
第 X 页：页面标题

页面正文内容……
```

最终 Word 中不得出现“页面类型：”“页面正文：”“资料依据：”或“阶段判断：”。

## 5. 页面标题规则

页面标题必须是观点型标题。

标题应该直接表达本页判断，而不是普通名词。

好的标题：

- 当前问题不是产品太多，而是缺少清晰的增长主线
- 消费者真正能记住的是具体状态，而不是企业资质
- 竞品普遍在讲专业，但没有形成可传播的消费词

不好的标题：

- 客户情况
- 产品分析
- 竞品介绍
- 行业分析
- 机会点说明

## 6. 页面正文规则

页面正文必须像 PPT 页面脚本，不得像长报告。

要求：

1. 每页只讲一个核心观点；
2. 正文优先使用 2-4 条结构化要点；
3. 每页正文原则上控制在 150-250 字；
4. 不写长段小作文；
5. 不在一页里塞入多个复杂判断；
6. 分析页要有收束判断，但最终 Word 不写“阶段判断：”固定标签；
7. 数据页要有“这说明什么”；
8. 表格页必须服务一个明确结论。

最终 Word 的正文要面向 PPT 制作，语言应更短、更直接、更有画面感。避免“资料显示 / 从现有资料看 / 本报告将 / 本建议书不会 / 阶段判断 / 后续需要 / 该页面说明 / 本章形成 / 当前资料中出现”等内部分析口吻。

最终 Word 页面正文不得大量重复：

```text
一句核心判断
1. ……
2. ……
3. ……
4. ……
阶段判断：……
```

同一章节内，连续页面不得都使用“一句话 + 编号列表 + 阶段判断”的格式；连续 3 页出现类似结构时必须主动改写。

## 6.1 页间逻辑检查

逐页内容不能只是一页一页孤立生成。每一页必须同时满足：

1. 承接上一页：本页要回答上一页留下的问题、延续上一页的判断，或推进上一页的结论。
2. 引出下一页：本页结尾或阶段判断要自然引出下一页要讨论的问题。

每生成一页时，Agent 需要内部检查：

- 上一页讲了什么；
- 本页承接了什么；
- 本页推进了什么；
- 下一页为什么应该出现。

这些检查不得作为固定字段写入最终 Word。最终页面只呈现自然的汇报逻辑，不写“承接上一页 / 引出下一页”等内部说明。

页面逻辑应形成：

上一页结论
→ 本页进一步拆解 / 验证 / 推进
→ 下一页需要继续回答的新问题

不得出现每页像独立笔记、上下页之间跳跃、前一页刚讲问题后一页突然给结论、没有铺垫就进入机会点、没有机会点确认就进入产品承接。

每页应是汇报故事线中的一个镜头，形成：

问题出现
→ 问题被拆开
→ 外部环境验证
→ 竞品格局对照
→ 企业自身回看
→ 机会点浮现
→ 战略承接落地

最终 Word 不机械写“承接上一页 / 引出下一页”，只通过正文语言自然连贯。

## 6.1.1 页面表达多样性规则

最终 Word 不应几乎全部由分析页、结论页、路径页构成。每一章应根据内容需要穿插目录页、章节页、反问页、金句页、分析页、数据页、对比页、表格页、路径页、总结页、转折页、关键判断页。

要求：

1. 连续 4-5 页密集分析后，应考虑加入过渡页、金句页、反问页、方向页或总结页；
2. 从“问题诊断”进入“竞争环境”前，应有承接页；
3. 从“竞争环境”进入“机会洞察”前，应有转折页；
4. 从“机会洞察”进入“战略承接”前，应有承接页；
5. 不要整份建议书都像分析笔记。

参考巧焙乐样板报告时，只学习页面表达逻辑，不复制样板报告的数据、行业内容、客户内容、品牌结论和文案。

## 6.2 客户可见第一视角表达规则

最终 Word 是给客户汇报的页面脚本，不是内部分析日志。客户版汇报中，“客户”应替换为具体品牌名、企业名或“品牌自身”。

不得使用：

- 客户当前的问题是……
- 客户资料显示……
- 客户希望……
- 客户当前的核心诉求……
- 该客户……
- 本客户……

应改为：

- 御棠源当前的问题是……
- 从御棠源现有资料看……
- 御棠源希望解决的是……
- 当前品牌增长的核心矛盾是……
- 对御棠源而言……
- 本项目需要回答的是……

如果是其他品牌，自动替换为对应品牌名称、企业名称或“品牌自身”。

## 7. 章节边界规则

不同章节只回答本章该回答的问题。

项目概况：
只讲项目背景、客户诉求、问题缘起、研究路径。
不得提前输出机会点、大单品候选、战略承接结论。

问题诊断：
只讲表层问题、零散问题归因、表面问题与根本问题。
不得输出具体机会点和产品承接。

环境分析：
只讲行业、消费者、竞品、企业自身。
不得输出客户最终机会点和大单品候选。

机会洞察：
正式输出四维交叉分析、具体机会点、机会点具体性检查和最终机会点清单。

战略承接：
在机会点确认后，输出产品承接、渠道承接、传播承接和推进节奏。

## 8. 资料依据规则

资料依据用于研究分析、过程审核版和内部检查，不进入最终 Word / PPT 制作版。过程审核版资料依据优先引用：

1. 客户资料；
2. 品牌手册；
3. 产品手册；
4. 官网 / 官方资料；
5. 行业报告；
6. 政策文件；
7. 消费者调研资料；
8. 竞品官网 / 旗舰店 / 公开资料；
9. 研究分析层中的资料来源。

不得把以下内容作为客户版页面主要资料依据：

- 确认点 1；
- 确认点 2；
- 确认点 3；
- 用户反馈；
- Skill 规则；
- AGENTS.md；
- MEMORY.md；
- agent_memory；
- 项目执行记录。

如果需要引用内部过程判断，可写：

“依据前序研究分析。”

但不得在客户版资料依据中写“依据确认点 2 用户反馈”。

如需保留最终资料依据，应单独生成内部文件，例如 `outputs/资料依据清单.md`。不得把资料依据写入客户可见 Word。

## 9. 信息释放顺序

逐页稿必须按以下顺序释放信息：

客户现状与问题
→ 行业环境
→ 消费者证据
→ 竞品证据
→ 企业自身证据
→ 机会点
→ 战略承接
→ 推进节奏

不得在前文提前泄露后文章节的结论。

## 10. 页数规则

完整建议书原则上控制在 50-60 页。

少于 50 页，说明论证可能不足。
超过 60 页，说明需要压缩合并。

但不得为了页数牺牲核心论证。

## 11. 暂停确认规则

每完成一个客户可见一级目录对应部分，必须暂停。

暂停时输出：

```markdown
本部分到此暂停。
请确认是否通过。
通过后可回复：第 X 部分通过，继续第 X+1 部分。
```

不得自动连续生成多个一级目录。

## 12. Word 生成规则

最终生成 Word 文档时：

1. 优先使用 python-docx 直接生成 .docx；
2. 不默认调用 LibreOffice；
3. 不默认调用 soffice；
4. 不打开本地 Office / LibreOffice 图形界面；
5. 如果无法直接生成 docx，应先保留 Markdown 或文本版完整内容，再提示用户处理。

生成 Word 前，必须做一次“PPT制作稿清洗”。

必须删除：

1. 页面类型；
2. 页面正文标签；
3. 资料依据；
4. 资料依据标签；
5. 阶段判断标签；
6. 本批生成内容；
7. 本部分对应客户可见一级目录；
8. 本章任务；
9. 本章不回答；
10. 本章预计页面；
11. 本部分到此暂停；
12. 请确认是否通过；
13. 通过后可回复；
14. 当前阶段说明；
15. Agent 执行说明；
16. 质量检查报告；
17. 总体检查结论；
18. 是否通过 Skill 规则检查；
19. 任何给用户看的操作提示；
20. 任何内部审核说明。

必须替换：

1. “客户” → 具体品牌名 / 企业名 / 品牌自身；
2. “客户资料显示” → “从现有资料看”或“从具体品牌现有资料看”；
3. “客户当前问题” → “具体品牌当前问题”；
4. “客户版资料依据”等内部表达不得进入正文。

必须优化：

1. 是否大量重复“一句话 + 编号列表 + 阶段判断”；
2. 是否有过多“资料显示 / 从现有资料看 / 本报告将 / 本建议书不会 / 后续需要 / 该页面说明 / 本章形成 / 当前资料中出现”等内部分析口吻；
3. 是否缺少过渡页、金句页、反问页、方向页或总结页；
4. 是否每页都能自然承上启下；
5. 是否更像 PPT 页面，而不是研究报告。

质量检查报告只能在对话中单独输出，或另存为独立文件（例如 `outputs/质量检查报告.md`），不得写入最终 Word 文档末尾。最终 Word 的最后一页应是客户可见的总结页、下一步建议页或战略推进页。

最终交付时应分成两个结果：

1. 客户可见 Word / PPT 制作版：只包含页码、页面标题和正文内容；
2. 内部质量检查：在对话中输出，或另存为独立 Markdown 文件。

## 12.1 输出模式区分

过程审核版用于对话分批生成，可以保留页面类型、资料依据、本章任务、本章不回答、本章预计页面，方便用户审核 Agent 是否按规则执行。

最终 Word / PPT 制作版用于交给 PPT 制作方，必须删除页面类型、资料依据、本章任务、本章不回答、本章预计页面、质量检查报告和内部过程说明，只保留页码、页面标题和正文内容。

## 13. 本阶段禁止事项

禁止：

- 确认点 3 未通过就生成逐页脚本；
- 默认按每 10 页一批输出；
- 跨章节提前释放后文结论；
- 项目概况提前输出机会点；
- 项目概况提前输出大单品候选；
- 环境分析阶段直接输出客户最终机会点；
- 机会洞察前提前输出战略承接；
- 用确认点、用户反馈或 Skill 规则作为客户版页面主要资料依据；
- 页面正文写成长篇报告；
- 金句页写多句解释；
- 反问页写多句解释；
- 最终生成 Word 时默认调用 soffice；
- 把页面类型、页面正文标签、资料依据、资料依据标签写入最终 Word；
- 在最终 Word 中写“阶段判断：”固定标签；
- 把本章任务、本章不回答、本章预计页面等过程控制信息写入最终 Word；
- 把质量检查报告写入最终 Word；
- 在最终 Word 中使用“客户资料显示”“该客户”“本客户”等内部分析口吻；
- 生成上下页互不承接的独立笔记式页面；
- 在同一章节连续 3 页使用“一句话 + 编号列表 + 阶段判断”的机械格式；
- 让整份建议书缺少过渡页、金句页、反问页、方向页或总结页。
